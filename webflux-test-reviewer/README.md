# WebFlux Test Reviewer - Agent Skill

Skill para evaluar pruebas técnicas de WebFlux de múltiples candidatos automáticamente.

## Installation

```bash
npx skills add https://github.com/jheisonmb/skills --skill webflux-test-reviewer
```

## What It Does

This skill guides an AI agent to:
1. Read the technical assessment statement
2. Detect all candidate directories
3. Evaluate each candidate's code against defined criteria
4. Generate an `EVALUATION.md` report for each candidate with:
   - Criterion-by-criterion evaluation
   - Interview questions based on code findings
   - Overall conclusion

## Expected Project Structure

```
assessment-project/
├── statement/
│   └── enunciado.md            # The technical assessment statement
├── jonathan-camano/
│   ├── (cloned candidate repo)
│   └── EVALUATION.md           # ← Generated by this skill
├── nicolas-ruge/
│   ├── (cloned candidate repo)
│   └── EVALUATION.md           # ← Generated by this skill
└── ...
```

## Evaluation Criteria

| Criterion | What's Evaluated |
|---|---|
| Readme | Exists, explains how to run. Penalizes AI-generated verbose readmes |
| GIT | Commit standard, atomic commits, branches/PRs |
| Webflux | Correct operator usage, no blocking, reactive error handling |
| Test Unitarios | Existence + coverage |
| Arquitectura | Pattern identification (Clean, Hexagonal, MVC) |
| Clean Code | Naming, SRP, no dead code, no unused imports |
| Manejo de Excepciones | Custom exceptions + handler + reactive operators |
| IaC | Existence + resource pertinence |
| DockerFile | Multi-stage build (compiles, not just copies JAR) |
| Docker-compose | Exists, runs app + DB |

Cross-cutting (reported if found):
- AI misuse detection
- Exposed secrets/credentials

## Usage

With the skill installed, navigate to your assessment project and ask the agent:

```
Evalúa las pruebas técnicas de los candidatos
```

or

```
Review the technical assessments
```

The agent will:
1. Read `statement/` to understand requirements
2. Find all candidate directories
3. Evaluate each one (in parallel if sub-agents are supported)
4. Generate `EVALUATION.md` in each candidate's directory

## Output Example

```markdown
# Jonathan Camano

## BACK:
• Readme: Si hizo buen readme e incluyo pdf
• GIT: Si presenta varios commits pero sin un manejo de estandar
• Webflux: Si uso programación reactiva
...

## Preguntas:
• Por que no manejar variables de entorno con el docker-compose?
• Diferencia entre map y flatmap

## Conclusión:
Es una buena prueba aunque debe fortalecer varios puntos
```
